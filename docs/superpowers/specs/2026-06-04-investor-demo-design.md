# 猫小伴投资方互动 Demo 设计

日期：2026-06-04

## 目标

为猫小伴制作一个可在本地浏览器打开的互动 demo 页面，并配套一份录屏脚本。这个 demo 面向投资方，目标不是完整替代真实 App，而是在 3 分钟内讲清楚：

- 猫小伴是什么。
- 它已经具备哪些真实产品能力。
- 它和普通桌面宠物、提醒工具、效率工具的差异。
- 为什么它有继续做成消费级陪伴产品的潜力。

页面必须同时适配桌面端和移动端。桌面端用于录屏和正式展示，移动端用于用户随时在手机上给投资方快速演示。

硬性表达要求：

- Demo 不能做成纯文字介绍页。
- 首屏必须用动画或可见演示效果，让人一眼看懂这是一个 macOS 桌面陪伴 App。
- 每个核心章节至少包含一种具体演示效果：宠物动作、桌面窗口避让、提醒气泡交互、状态流程动画、录屏模式自动推进等。
- 文案只能辅助解释，不能成为页面的主要信息载体。

## 核心判断

猫小伴的投资方 demo 不应该做成传统官网，也不应该只堆功能截图。更合适的形态是“互动路演页”：用户打开页面后，可以像翻一份精炼 pitch deck 一样逐屏推进；点击“开始演示”后，页面按录屏节奏自动播放，方便录制视频。

这个形态有三个好处：

- 比静态 PPT 更像产品。
- 比真实 App 实录更容易控制叙事。
- 比概念宣传片更可信，因为页面中的功能来自当前仓库已有能力。

## 受众

主要受众：

- 投资方。
- 潜在合作方。
- 对消费级 AI / 桌面陪伴产品感兴趣的人。

受众不会默认理解 macOS 桌面宠物的技术细节。因此页面文案要把技术能力翻译成用户价值，例如：

- “前台应用分类”表达为“知道你现在是在工作、会议还是休闲”。
- “窗口覆盖率判断”表达为“尽量不挡你的工作窗口”。
- “本地读取系统状态”表达为“不上传屏幕内容，不联网同步数据”。

## 产品定位

一句话定位：

> 猫小伴是一个懂工作状态、低打扰、本地隐私优先的 macOS 桌面电子宠物。

辅助说明：

> 它不是一个单纯会动的小猫，而是一个轻量的桌面陪伴系统：工作时安静，休闲时活跃，夜晚会睡觉，需要时轻轻提醒你喝水、休息和久坐。

## 页面结构

### 1. Hero：一句话讲清楚产品

内容：

- 产品名：猫小伴
- 定位文案。
- 蓝双布偶猫视觉形象。
- 两个入口按钮：
  - 开始 3 分钟演示。
  - 查看功能亮点。

体验要求：

- 首屏必须直接看到猫的形象，不只是在标题或导航里出现名字。
- 首屏必须有可见动效，例如猫小伴轻微呼吸、眨眼、在桌面边缘停留，或从窗口边缘探出。
- 桌面端首屏要露出下一屏一点内容，暗示可以继续滚动。
- 移动端首屏要控制信息密度，优先让投资方一眼理解“这是一个桌面陪伴产品”。

### 2. 场景：它怎样陪你工作

展示工作状态下的价值：

- 猫在桌面边缘安静陪伴。
- 自动贴边，尽量不遮挡窗口。
- 工作时降低动作频率。
- 全屏 / 演示场景下低打扰。

页面表现：

- 用一个模拟 macOS 桌面窗口展示猫小伴停在边缘。
- 点击或自动播放时，猫从轻微眨眼 / 趴着状态切换到安静陪伴。
- 前台窗口轻微移动或扩张时，小猫位置保持在安全边缘，直观看出“低打扰”和“尽量不挡窗口”。

### 3. 场景：它怎样在休闲时活起来

展示休闲和空闲状态：

- 空闲一段时间后，小猫会散步、探头、伸懒腰。
- 北京时间日夜节律：白天陪伴，晚上睡觉。
- 视觉动作来自当前资产库，而不是临时概念图。

页面表现：

- 展示 walk-left / walk-right、peek、stretch、sleep 等动作缩略预览。
- 移动端用横向滑动卡片展示动作，不强行塞满一屏。
- 动作预览必须使用真实 GIF、帧动画或 CSS 驱动的序列切换，不能只放静态图片加说明。

### 4. 功能：提醒系统不是闹钟

展示提醒能力：

- 喝水、休息、久坐、睡觉提醒。
- 提醒气泡支持完成、稍后、跳过、关闭。
- 安静工作、会议、全屏时会延后提醒。
- 延后队列恢复后再出现。

页面表现：

- 用一个提醒气泡模拟交互。
- 提供“完成”“稍后”“跳过”三个按钮作为演示动作。
- 文案强调“轻提醒”，避免像效率工具或健康管理软件那样强压迫。
- 点击“稍后”后，气泡进入延后队列；点击“完成”后，猫出现轻微正反馈动作。

### 5. 能力：为什么它不是普通桌宠

展示底层能力，但不堆工程术语：

- 状态识别：工作 / 休闲 / 睡觉 / 空闲。
- 应用分类：工作、休闲、沟通/会议、中性。
- 窗口避让：根据前台窗口覆盖率决定行为。
- 行为状态机：每次动作有原因。
- 设置持久化：大小、位置、模式、提醒等可保存。
- 调试面板：能观察状态、动作、应用分类、提醒队列。

页面表现：

- 用“桌面信号 -> 行为判断 -> 小猫动作”的三段式流程图。
- 右侧展示轻量调试面板 mock，不直接暴露过多原始字段。
- 流程图需要有自动高亮或步骤推进效果，让观众看到系统如何从桌面状态推导出宠物行为。

### 6. 隐私：本地优先

展示当前隐私边界：

- 读取前台应用名称和 Bundle ID。
- 读取前台窗口几何信息。
- 读取用户空闲时间。
- 本地保存提醒和设置。
- 当前版本不上传屏幕内容，不读取窗口正文内容，不联网同步数据。

页面表现：

- 用清晰的“读取 / 不读取 / 不上传”分组。
- 语气要克制，避免夸张承诺。

### 7. 投资亮点：长期价值

展示商业和产品想象：

- 情绪陪伴：从工具型软件进入日常陪伴场景。
- 桌面入口：长期停留在用户工作流里。
- 个性化：宠物形象、动作、提醒、情绪反馈可扩展。
- 商业化方向：高级皮肤、动作包、AI 对话、健康提醒、品牌联名、企业办公陪伴。

页面表现：

- 用 4 张短卡片表达，不写成长商业计划书。
- 避免承诺尚未实现的能力为当前功能，必须标明“可扩展方向”。

### 8. 录屏模式

页面提供一个“开始演示”按钮，进入录屏模式：

- 自动切换 6 个章节。
- 每个章节停留 15 到 30 秒。
- 页面显示当前章节标题和一句核心旁白。
- 允许暂停、上一段、下一段。
- 录屏模式总时长控制在 3 分钟左右。

录屏模式章节：

1. 产品定位。
2. 工作陪伴。
3. 智能提醒。
4. 状态感知和隐私。
5. 设置与可调试能力。
6. 商业化想象。

## 视觉方向

整体气质：

- 温暖，但不幼稚。
- 可爱，但不低龄。
- 像一个有产品质感的消费级软件 demo，而不是宠物小游戏官网。

建议视觉语言：

- 背景保持干净，使用浅色或接近真实桌面的中性色。
- 保留猫小伴的蓝双布偶猫识别度。
- 使用真实动作帧、GIF 或 spritesheet 作为主要视觉资产。
- UI 元素控制圆角，避免过度卡片化。
- 避免整页只使用单一蓝紫色调。

## 移动端要求

移动端不是桌面页的压缩版，而是独立优化的展示路径：

- 一屏一个重点。
- 固定底部或顶部提供章节导航。
- 按钮和交互区域适合手指点击。
- 关键文字不超过 2 到 3 行。
- 动作预览使用横向滑动或单列卡片。
- 录屏模式在手机上也能手动推进。

最低适配宽度：

- 360px 移动端。
- 768px 平板。
- 1280px 桌面。

## 录屏脚本结构

目标时长：约 3 分钟。

| 时间 | 内容 | 画面 |
| --- | --- | --- |
| 0:00-0:20 | 猫小伴是什么 | Hero 与猫形象 |
| 0:20-0:55 | 工作时安静陪伴 | 模拟桌面、贴边、低打扰 |
| 0:55-1:30 | 轻提醒系统 | 提醒气泡、稍后、延后 |
| 1:30-2:05 | 状态感知与本地隐私 | 信号流程图、隐私分组 |
| 2:05-2:35 | 设置和调试能力 | 设置 Tabs、调试面板 mock |
| 2:35-3:00 | 投资亮点和下一步 | 商业化方向卡片 |

旁白风格：

- 克制、清楚、可信。
- 不使用夸张 AI 概念。
- 先讲用户价值，再讲技术能力。

## 技术方案

实现形态：

- 新建一个本地静态 demo 页面。
- 使用原生 HTML、CSS、JavaScript，避免引入构建系统。
- 页面可通过本地文件直接打开，也可用简单本地服务预览。

建议路径：

```text
investor-demo/
  index.html
  styles.css
  script.js
  assets/
  recording-script.md
```

资源复制原则：

- 只复制 demo 必需素材到 `investor-demo/assets/`。
- 保留原始素材路径说明。
- 不移动、不改名原产品素材。

## 验收标准

页面体验：

- 桌面浏览器打开后首屏能看出产品名、猫形象和定位。
- 首屏不阅读正文也能判断：这是一个会在桌面边缘陪伴用户、尽量不打扰工作的 App。
- 至少 5 个核心章节包含动画或交互演示，不允许只有静态文案卡片。
- 手机宽度下没有文字重叠、按钮溢出或图片遮挡。
- 主要章节可通过导航或滚动访问。
- 录屏模式可以按章节自动推进，也可以手动控制。

内容准确性：

- 页面展示的当前功能必须能在项目 README 或版本文档中找到依据。
- 未来方向必须标注为可扩展方向，不能写成已完成能力。
- 隐私描述必须和 `README.md` / `PRIVACY.md` 保持一致。

交付物：

- 可打开的互动 demo 页面。
- 移动端适配。
- 3 分钟录屏脚本。
- 简短使用说明。

验证：

- 静态文件结构检查。
- 桌面端截图检查。
- 移动端截图检查。
- 关键按钮和录屏模式交互检查。

## 非目标

本轮不做：

- 改造真实 macOS App 功能。
- 接入真实后端。
- 公开发布或部署。
- 编写正式融资 BP。
- 承诺 Apple Developer ID、Notarization 或 GitHub Release 完成。

## 风险与处理

风险：页面过于像官网，投资方看不到产品已经能跑。
处理：首屏和核心章节必须使用当前真实猫动作资产，并明确展示桌面场景。

风险：页面过度宣传未来 AI 能力。
处理：当前功能和未来方向分区表达，文案上明确区分“已具备”和“可扩展”。

风险：手机端信息过密。
处理：移动端一屏只讲一个卖点，长信息进入折叠或下一屏。

风险：真实 App 截图不足。
处理：先用项目已有动作 GIF 和模拟桌面表达，后续如需要再补真实录屏素材。
